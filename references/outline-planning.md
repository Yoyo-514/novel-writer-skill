## Purpose

The outline stage turns the blueprint into a medium-scale plan. Focus on structure, main conflict chains, character turns, emotional flow, information reveals, and module transitions. Do not overplan every scene here; that belongs to chapter outlining.

## Planning Principles

- Let the blueprint decide priorities.
- Apply any selected topic/style profile from `topic-guidance.md` when deciding module goals, reader rewards, pacing, and scene emphasis.
- Treat the finished outline as a strong reference for drafting, not disposable brainstorming notes.
- Allow outline changes during writing only when the user explicitly requests a plan change or approves a proposed adjustment.
- For long works, use modules/volumes as the main unit.
- Plan near modules in detail and far modules lightly.
- Keep a brief whole-story skeleton as a map, not a prison.
- Treat modules as rising waves: each should deepen conflict, character, world, or meaning rather than merely repeat.
- Fit plot movement and emotional experience together: setup builds pressure, climax releases or transforms it, transition processes consequences and points forward.
- Use information gaps and chapter/module-end questions strategically to maintain expectation.

## Existing Material Intake

When the user provides notes, characters, setting, or partial plot:

1. Summarize what is already usable.
2. Identify gaps that block outline work.
3. Keep base character and setting details minimal unless they directly affect the current module.
4. Ask where to start: whole skeleton, first module, repair pass, or expansion pass.

## Missing Outline Proposal Gate

If the blueprint, whole-story direction, or active module premise is missing, empty, or too vague, do not write a full outline immediately. First offer 2-3 outline directions and wait for user confirmation.

Each option should include:

- core story direction
- likely module/volume shape
- main character or relationship pressure
- reader reward
- biggest risk or missing assumption

After the user chooses, merges, or modifies an option, turn the selected direction into `OUTLINE.md` or the requested module outline.

Create or update `novel/OUTLINE.md` with a lightweight global map:

```markdown

- Core vision:
- Reader promise:
- Main story engine:

### Module 01 - [Name]
- Role in the story:
- Starting state:
- Main conflict:
- Character movement:
- Key reveal:
- Climax:
- Ending state / next hook:

### Module 02 - [Name]
...

## Ending Direction
```

Do not force a fixed endpoint too early. If the story is designed for long-form expansion, record the current ending direction and allow future modules to extend.

## Completion Check

Before moving to chapter outlining, check:

- The module has a clear job in the larger story.
- The main conflict chain has cause and effect.
- Character change has event support.
- The emotional high point is identifiable.
- Reveals and withheld information serve expectation, not confusion.
- Setting details are necessary and interact with characters.
- The module is the right size: not a tiny transition inflated into a volume, and not too large to control.

## Mid-Drafting Outline Changes

Plans often fail to survive contact with actual prose. Support iteration, but keep it explicit:

- Do not rewrite `OUTLINE.md` just because a drafted scene drifted.
- If prose no longer fits the outline, first try to revise the prose back toward the plan.
- If the user says the new direction is better, switch to outline revision mode.
- Before revising the outline, summarize:
  - what changed
  - why the change may improve the story
  - which later modules, chapter outlines, character arcs, information reveals, and timeline entries are affected
  - whether completed chapters need revision or can remain as-is
- After confirmation, update the outline and note the change.

## Suggestion Pattern

When offering options, use numbered option IDs such as `[1.1]`, `[1.2]`. Briefly explain why each might work, then remind the user that options can be merged.
